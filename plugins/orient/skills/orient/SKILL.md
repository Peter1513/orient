# Orient Skill

## 命令介面

| 命令 | 含義 |
|---|---|
| `/orient` | 基本啟動：`scan:off`，`search:auto` |
| `/orient scan:on` | 啟動前先做 repo scan |
| `/orient search:always` | 每輪 brainstorming 後必搜先例 |
| `/orient search:auto` | Agent 自判是否搜尋（預設） |
| `/orient search:off` | 不做 web 搜尋 |
| `/orient scan:on search:always` | 參數可任意組合，以空格分隔 |

**解析規則：**
- 未指定 `scan`：預設 `scan:off`
- 未指定 `search`：預設 `search:auto`
- 多參數無順序要求
- 非 `key:value` 格式之自由文字（如 `/orient 我想做 X`）：視為初始想法，帶入接收初始想法節
- 不認識之 `key:value` 參數（如 `scan:true`）：忽略並使用預設值，繼續執行，不報錯

## 接收初始想法

啟動後，若 user 尚未提供初始想法，詢問：

> 「請描述你想做的事。不需要完整——幾句話說明你的目標或想解決的問題即可。」

若 user 在 `/orient` 命令中已附帶說明（如 `/orient 我想做 X`），或在同一訊息內命令之後附有初始想法，直接以此為初始想法，不重複詢問。

取得初始想法後：
1. 若 `scan:on`：執行元件一 Repo Scan
2. 若 `scan:off`（預設）：直接進入元件二 Brainstorming Engine

## 元件一：Repo Scan（選用）

**觸發：** `scan:on` 參數存在時執行；否則略過此節，直接進元件二 Brainstorming Engine。

**掃描順序：**
1. 讀 `CLAUDE.md` / `AGENTS.md`（若存在）——取 agent 規約、禁令、工法偏好
2. 讀 repo-local skill / instruction 檔（若存在）
3. 依上述文件指引，決定其他相關檔範圍；若無指引，掃 `README.md`、`docs/`（若存在）

若 `CLAUDE.md` / `AGENTS.md` 不存在，逕行步驟 2-3。

**帶入 brainstorming 之輸出（Scan 摘要）：**

```
【Repo Scan 摘要】
規約重點：[條列規約與禁令，或「無 CLAUDE.md / AGENTS.md」]
現有相關元件：[條列 skill / instruction 檔；若有衝突，註明；或「無」]
未察區域：[列出未讀之目錄 / 檔案，或「全域已掃」]
```

此摘要作為元件二 Brainstorming Engine 第一輪之背景 context。

## 元件二：Brainstorming Engine

### 步驟 A：領域偵測

取得初始想法（與 Repo Scan 摘要，若有）後，問：

> 「這個任務的最終交付物是 code / 可執行系統，還是文件 / 決策 / 流程設計？」

| 回答 | 啟用語境 |
|---|---|
| code / 可執行系統 | **軟體開發語境**：問題框架含架構、元件邊界、資料流、測試方式、部署限制 |
| 文件 / 決策 / 流程 | **非軟體語境**：問題框架含目標成果、工作流程、涉及人員、驗收方法（不問架構） |

若 user 答案模糊，追問：「完成後，你交出的東西主要是給電腦跑的，還是給人讀 / 用的？」

若 user 答案兼含兩類，追問：「主要交付物優先：(1) 可執行的程式碼，還是 (2) 流程 / 文件方案？優先的那項決定問題框架。」

**範疇說明：**
- 軟體語境涵蓋所有「最終產物含可執行代碼」的任務（含 IaC、腳本、SQL migration、Jupyter notebook）
- 非軟體語境涵蓋「交付決策 / 設計 / 文件」的任務（縱使過程涉及程式）

領域偵測完成後，明述：「確認：我們走 [軟體開發 / 非軟體] 語境。繼續嗎？」取得 user 任意確認（yes、好、OK、或直接繼續補充資訊）後，進入步驟 B；若 user 未表示異議即繼續提供資訊，視為確認。偵測後不再切換。

### 步驟 B：Q&A 迴圈

若前步驟已執行 Repo Scan，回顧其摘要（規約重點、現有相關元件）作為背景，勿重複詢問已知資訊。

**規則：**
- 一次問一個問題；等 user 回答後再問下一題
- 每輪問答後，依 `search` 參數觸發元件三 Web 搜尋整合
- 帶先例發現入下一輪問題 context

---

**軟體開發語境問題集：**

1. 此功能的主要使用者是誰？他們最核心的需求是什麼？
2. 成功的定義是什麼？有哪些可量測的指標？
3. 有哪些技術限制或既有系統必須相容？
4. 邊界在哪裡——哪些明確不在此次範疇內？
5. 若此功能三個月後被棄用，最可能的原因是什麼？

---

**非軟體語境問題集：**

1. 這個任務的目標成果是什麼？完成後交出的是什麼？
2. 誰會使用或受影響？他們目前如何應對這個問題？
3. 成功的樣子是什麼？如何確認目標達成？
4. 哪些範疇或步驟明確不在此次工作內？
5. 若三個月後此成果被棄用，最可能的原因是什麼？

---

**收斂判準（每輪後評估）：**

> 目標、範疇邊界、成功訊號三者是否均已有明確答案（含邊緣情況），足以讓人在不追問的情況下逐步寫出工作計畫？

- 是 → 進入輸出階段（先輸出 B，再回填輸出 A 之先例來源標記；輸出 A 其餘部分可先行填入）
- 否 → 繼續問答（問題集用完後，針對仍不清晰的目標 / 範疇 / 成功訊號追問；每個模糊點最多追問 2 次；無新進展時視為收斂）

---

**異常處理：**

- 答案含混：「能舉一個具體例子嗎？或給一個可以被證偽的標準？」
- 答案矛盾：「你先說 [A]，後說 [B]，兩者不相容——請選一個。」
- User 明確停止（說「夠了」「直接輸出」等）：立即進入輸出階段，以當前所得輸出（同上，先輸出 B，再回填輸出 A 先例來源標記）

## 元件三：Web 搜尋整合

每輪 Q&A 後，依 `search` 參數觸發：

| 參數 | 行為 |
|---|---|
| `search:always` | 每輪必搜，無例外 |
| `search:auto` | 若當輪問答出現「此輪新引入」的技術名 / 方案名 / 可命名問題領域或實踐名稱（指有行業共識名稱者，如「transactional email」、「daily standup」；不含純量測指標或通用形容詞）（非前輪已搜過），則搜；否則略過 |
| `search:off` | 整個 orient session 不做 web 搜尋 |

**觸發後動作：**
1. 以當輪關鍵詞搜尋相關先例；若當輪無可命名關鍵詞（如純描述性回答），以「[初始想法核心詞] + [本輪問題主題]」組合為搜尋詞
2. 整理發現：各先例名稱、一句簡述、與當前目標之主要差異
3. 帶入下一輪：帶最相關的 1-2 個先例（若已收斂則僅記入輸出 A 素材池），格式：「找到相關先例：[X]，其主要差異為 [Y]。這影響你的想法嗎？」其餘先例僅記入輸出 A 素材池
4. 記錄至輸出 A 素材池（最終輸出時整理成報告）

**無相關結果時：**
輸出固定句：「已搜尋：[搜尋詞]。未找到相關先例。」此記錄亦寫入輸出 A 素材池。

## 輸出 A：先例參照報告

**產出順序：** 先產出輸出 B，再一併填寫輸出 A（已搜尋先例表、決策理由、先例來源標記），以確保決策理由與步驟編號一致。

**格式：**

---
**先例參照報告**

**已搜尋先例：**
（資料來源：元件三各輪搜尋所得；「與本目標差異」欄取自元件三步驟 2 之整理結果）

| 先例名稱 | 簡述 | 與本目標差異 |
|---|---|---|
| [名稱] | [一句話] | [具體差異] |

（若無相關先例：「搜尋詞：[X]。未找到相關先例。」）
（若 `search:off`：「本次未做先例搜尋（search:off）。」）
（若找到但不適用：仍列入表格，差異欄填「根本假設不符，不採用」）

**決策理由：**
（依據：對照 brainstorming 所得之成功訊號、限制、範疇邊界，判斷各先例之借鑑與棄用）
- 借鑑 [先例名稱] 之 [具體做法]，因符合 [成功訊號 / 限制]
- 棄用 [先例名稱] 之 [具體做法]，因與 [成功訊號 / 限制] 不符

（若無先例：此段省略）

**輸出 B 先例來源標記：**
（於輸出 B 完成後回填）
步驟 [N] 參考 [先例名稱] 之 [具體做法]

（若找到先例但均未借鑑：「搜尋到先例，惟均不採用；理由見上方決策理由欄。」）
（若無先例：「輸出 B 無先例借鑑。」）

---

## 輸出 B：完整計畫

**產出時機：** 收斂判準達成（或 user 明確停止）後，先於輸出 A 之先例來源標記。

**格式：**

---
**釐清後目標**

[一句核心陳述]

[說明 3-5 句，涵蓋：(1) 背景——誰、為何要做；(2) 範疇——含什麼、不含什麼；(3) 關鍵限制——取自 brainstorming 中成功訊號、限制、邊界答案]

**完整工作步驟**

步驟 N：[步驟名稱]
交付物：[具體產出物——文件、程式碼、決策、清單等，非「進行中狀態」]
驗證：[如何確認此步驟完成——可觀察的條件，非「感覺完成了」]

（依此格式列出所有步驟；步驟合力涵蓋從當前狀態到目標達成的完整路徑，不留後續待續工作）

---

**非軟體語境步驟範例：**

步驟 1：整理現有 code review 流程清單
交付物：包含現行所有步驟之文件，含每步責任人與時間估算
驗證：可對照實際操作逐項確認，無遺漏步驟

步驟 2：標記瓶頸點
交付物：標記後之清單，含每個瓶頸之症狀描述與發生頻率
驗證：每個瓶頸有可觀察的失敗徵兆，非主觀判斷

步驟 3：設計改進方案
交付物：改進後流程文件，含每個瓶頸對應之解決做法
驗證：改進做法可對應步驟 2 中每個瓶頸，無瓶頸未被覆蓋

**步驟粒度要求：**
- 每步有明確交付物（非「進行中狀態」）
- 每步有可驗證的完成條件（非「感覺完成了」）
- 若步驟描述超過 3 句仍不清晰，繼續拆分
